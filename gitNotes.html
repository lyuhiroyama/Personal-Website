Git (1)

  • VCS
    • (def)
      • (git-scm site) System that records changes to a file or a set of files over time
        so that you can recall sepcific versions later.
      • ‘Version Control Systems’
    • 3 Types of VCS
      • 1) Local Version Control Systems
        • (def) System which the local databse located on your local computer, where every file change is stored as a patch.
        • (e.g.) GNU RCS (Revision Control System)
      • 2) Centralized Version Control Systems
        • (def) System where one single server acts as the main centralized repository which stores every version of code.
          • The single central server here is a single point of failure. Any downtime/issues with the server can halt an entire team’s efforts.
        • (e.g.) CVS/Subversion/Perforce/ClearCase
      • 3) Distrubuted Version Control Systems
        • (def) System where every team member’s computer / client fully mirrors the entire repository, including its full history. Every clone is a full backup of all the data.
        • (e.g.) Git/Mercurial/Darcs/Bazaar/BitKeeper
    • 2 Methods of Managing changes to files
      • 1) Snapshot-based version control
        • (def) VCS’s that saves the entire state of a file/system as a single instance, known as a snapshot.
        • (e.g.) Git
        • Better suited for the distributed nature of software development.
      • 2) Delta-based version control
        • (def) VCS’s that saves only the changes made to a file/system rather than the entire state.
        • The change is stored as a delta Δ (Difference from the previous version).
          • Different states of the system (current or previous) is reconstructed by combining the preceding deltas.
        • (e.g.) Subversion VCS

  • Git
    • (def)
      • A free, open-source distributed version control system.
        • (scm) “Core of Git is a simple key-value data store. When you insert any kind of content into a Git repository, Git hands you back a unique key that you can use later to retrieve that content.”
        • The object’s file name (inside lyuhiroyama/yourRepo/.git/objects/) is a hash,
          and the actual content you are saving is the actual contents of the file.
      • Operates at both the local and remote level.
      • Developed in 2005 by Linus Torvalds.
    • On Data Integrity
      • (def) Git assures integrity of the data being stored by using checksums as identifiers.
        • *Checksums : Value returned from a one-way hash algorithm. Modifying the data itself (the code) will change the checksum value returned. Hence why it is used to validate the integrity of the data. It’s aka ‘fingerprints’.
        • Uses the SHA-1 cryptographic hash function.
      • Use of checksums are what keeps Git from losing information in transit or
        getting file corruption
        without Git being able to detect it.

    • 3 sections of a Git project (locally): (: modify/staged/commit)
      • 1) Working Directory/Tree (modify) (: check out)
        • (def) A directory on your local machine where you have cloned or “checked out”
          a version/branch of a .git repository.
        • It’s where you ‘modify’ your files.
        • Untracked files: Refers to what files in the working directories are.
      • 2) Staging Area (staged)
        • (def) A directory generally contained within the Git directory that stores information about what will go into your next commit.
        • Aka ‘index’.
        • To ‘stage’ : (def) To mark a modified file in the working directory to go in your next commit.
        • It is when file(s) are staged, that a checksum for each file is computed.
      • 3) .git directory (commit)
        • (def) Directory on your local machine, where Git stores the
          metadata &
          object database for your project.
          • *Object database: Git stores the code you commit in the form of an object.
            The code itself as the object value, and the checksum of the code as the object key.
        • To ‘commit’ : (def) To take the files as they are in the staging area, and stores that ‘snapshots’ (Revision History) permanently to your .git directory.
          • *Snapshot : Set of files/directories that represents an entire project of a particular moment.

  • Glossary (Terms)
    • Repository
      • (def) Directory containing all files/directories pertaining to your project, including the .git directory, working directory, & its additional metadata.
    • Commit
      • (def) Taking the staged changes and saving it to a local repository.
      • To take the files as they are in the staging area, and stores that ‘snapshots’ (Revision History) permanently to your .git directory.
    • Push
      • (def) To update a remote repository with the commits you made locally.
    • Standard Input / Output (stdin / stdout)
      • Standard Input:
        • (def) Input expected from the keyboard.
      • Standard Output:
        • (def) Output of he code, usually to the terminal window.
    • Revision History
      • (def) A chronological record of all changes/modifications made to a project over time within a version control system.
      • Aka ‘commit history’ / ‘version history’
    • Branch
      • (def) A representation of an independent line of development.
    • Branching
      • (def)(scm) To diverge from the main line of development and continue to do work without messing with that main line.
      • Branching in Git is considered lightweight and instantaneous compared to other VCS.
    • Main branch (: default branch)
      • (def) The default branch that is displayed when anyone visits your repository.
      • GitHub names all default branches ‘main’ by default for any new repositories.
    • Feature branch
      • (def) A branch that is branched off of the default branch to build a feature with,
        in isolation from changes that other people are making to the repository.
    • Cloning
      • (def) To copy an existing Git repository into a new local directory.
    • Tracked Files
      • (def) Any files in a Git repository that is ‘tracked’, by either being staged/committed/pushed.
      • *Untracked Files : Any files in the working directory (within a Git repository) that have not yet been staged (git add).
    • Root commit
      • (def) The initial commit. The ver first commit in a Git repository.
    • Blobs/Trees/Commit/Tag (Objects)
      • (def)
        • There are 4 types of objects used in Git: Blobs/Trees/Commit/Tag.
        • In short:
          • Blob: Contents of a file.
          • Tree: A directory listing (of blobs and trees).
          • Commit: A snapshot of the working tree.
      • Blobs
        • (def) The object type used to store the contents of each file in a repository.
          • Prety much a basic key-value object.
          • Key: The hash/checksum generated with SHA-1, using the content you are saving as the input for the hash.
          • Value: The content itself.
          • Only stores the file’s contents. (No metadata).
        • ‘Binary Large Object’
          • ‘Large’ refers to the ability of this data type to handle substantial amount of binary data.
        • You can find blobs in the …/yourRepo/.git/objects/ directory.
        • Blobs contain no information that directly links them ot a file path.
        • For further details see: git hash-object -w textFile.txt section.
      • Tree Objects
        • (def)
          • What links together a list of file blobs with their corresponding paths/names/permissions.
          • A directory listing (of blobs and trees).
          • Has its own hash, like blobs do.
        • Created from files/directories in the staging area.
        • Aka ‘Trees’
      • Commit Objects
        • (def) “A snapshot of the working tree”.
        • Object which includes:
          • A pointer to the main tree (Aka top-level tree / root directory).
            • *This is in the form of a tree object.
          • Meta-data such as the committer/commit message/commit time.
          • Parent commits, if any.
    • File modes (’100644’)
      • (def)
        • In Linux, every file has a file mode that represents the read/write/execute permissions that different users have. 5 different file modes seems to be used in Git.
        • (scm) “The modes are taken from normal UNIX modes.”
      • 100644
        • (def) A regular file. Non-executable.
      • 100755
        • (def) An executable file.
        • (e.g.) .exe/.py/.sh files.
      • 040000
        • (def) A Directory.
      • 120000
        • (def) A symbolic link. (A file that points to another file/directory.)
      • 160000
        • (def) A Submodule. (A Git repository within another Git repository.)
    • Plumbing / Porcelain commands (:low-level, high-level)
      • Plumbing commands
        • (def) The foundational/technical, ‘low-level’ commands/operations that deal with the internal mechanisms of Git, such as Git objects & trees.
          • Just like plumbing to do with pipes and valves. The behind-the-scene stuff.
        • (e.g.) git update-index --add someFile.txt
      • Porcelain commands
        • (def) The other user-friendly ‘high-level’ commands/operations.
          • Just like to do with porcelain fixtures (sinks, toilets). What the end-users interact with.
        • (e.g.) git add someFile.txt

    • git hash-object (: Blobs / How git stores info)
      • (def)
        • (man) Compute object ID and optionally creates an object from a file.
          • Primarily used for creating blobs. Could be used to create tree/commit/tag objects.
        • (scm) Computes the object ID value (key value for the object databse) for an object with specified type with the contents of the named file,

          and optionally writes the resulting object into the object database (using -w).
          • Outputs its Object ID value (key value).
          • Seems to use the SHA-1 algorithm to compute the object ID value (the hash) by default.
          • This command does NOT commit files.
        • *Object database : The .git/objects directory.
      • git hash-object textFile.txt
        • (def) Computes the object ID value (key value for the object database) for the content of textFile.txt.
          • *Without -w, it does this without storing the output in the object database.
        • Output (something like this):

          e69de29bb2d1d6434b8b29ae775ad8c2e48c5391
      • git hash-object -w textFile.txt
        • -w : Actually writes the object into the object database.
        • What it does:
          • Output (something like this):

            e69de29bb2d1d6434b8b29ae775ad8c2e48c5391
          • Creates a directory in yourRepository/.git/objects directory with the first two hexadecimal digits of the hash (e6) as the name.
          • Within the e6 directory is a file (Blob), named with the remaining characters of the hash (9de29bb2d1d6434b8b29ae775ad8c2e48c5391). This is where the original data of textfile.txt is present.
          • In this example:
            → Key:
            9de29bb2d1d6434b8b29ae775ad8c2e48c5391

            → Value: textfile.txt
      • echo ‘Test Content’ | git hash-object -w --stdin
        • --stdin : Tells git hash-object to get the content to be processed from standard input (that for shell being, from the keyboard). Otherwise, git hash-object expects a filename argument.
