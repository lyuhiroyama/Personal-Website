  • Value Types & Reference Type
    • (def) All Types in Swift are one of the two.
    • Value Types:
      • (def) A type that creates a new instance (copy) when assigned to a variable or constant, or when passed to a function.
      • (e.g.) Structs/Enums/Tuples/Int/Double/String/Array/Dictionary/Set
    • Reference Type:
      • (def) A type that once initialized, when assigned to a variable or constant, or when passed to a function, returns a reference to the same existing instance.
      • Once instantiated, when we either assign it or pass it as a value, we are actually assigning or passing around the reference to the original instance (i.e. its location in memory).
      • (e.g.) Class/Function/Closure

  • String Interpolation
    • (def) a way to construct a string from constants, variables, literals, and expressions.

  • Tuples
    • (def) a group of different values enclosed in parentheses. Each value inside a tuple can be of different data types.
    • Ordered. Duplicates allowed.
    • Fixed in size. Mutable, but values/data types can’t be added/removed. (If assigned to a var, that is. Immutable if on let.)
    • You can also give element names to each values: var kobe = (name: “Kobe”, number: 24)
    • Values inside can be accessed using dot notation with element names or indice.
    • Tuple is effectively just a Struct without a name.
      • Use tuples when you want to return two or more arbitrary pieces of values from a function. Structs when you have some fixed data you want to send or receive multiple times.

  • Error Handling (Throwing functions, do try catch, error propagation)
    • (def)
      • (def) Process of responding and recovering from error conditions.
      • The Throwing Function must catch all errors.
      • Error propagation:
        • When a throwing function throws (signals) an error, it then gets propagated to the scope from which it’s called (function within the do block in the example below.)
    • The process (try)
      • Step 1: Define possible errors that can happen with enums.
        • Essentially naming errors.
      • Step 2: Define a throwing function that can flag up error if they happen.
        • The function ‘throws’ an error to signal a problem.
      • Step 3: Call the throwing function and handle any errors that might happen, using do-try-catch.
        • Create a do block to contain the calling of the throwing function (try) and the catching (catch).
        • Call throwing function with a try expression (’try’ since it can thrown an error.)
        • Prepare the catch clauses to handle the appropriate errors.
    • try! (Disabling of Error Propagation)
      • (def) Expression used to disable error propagation when you are certain that a throwing function will not throw an error.
    • try? (Optional try)
      • (def) Converts errors to Optional Values.
        • Expression used to handle error by converting in to an optional value. If error is thrown, the optional value nil is returned. If not, optional value containing a value is returned.
      • An ‘optional try’.
      • Useful wen we don’t care what error was thrown, and only whether it throws an error or not.
      • Can be combined with nil coalescing to give it a default value if it returns a nil.
    • Code:
      //Step 1: Define errors
enum PasswordError: Error {
	case short, obvious
}

//Step 2: Define Throwing Function
func checkPW(_ password: String) throws -> String {
	if password.count < 5 {
		throw PasswordError.short	
	}
	if password == "password" {
		throw  PasswordError.obvious
	}

	if password.count < 8 {
		return "Fine"
	} else {
		return "Good"
	}
}

//Step  3: Run function, handle errors
let examplePW = "melo"

do {
	let rating = try checkPW(examplePW)
	print("PW rating: \(rating)")
} catch PasswordError.short {
	print("ERROR: Too short")
} catch PasswordError.obvious {
	print("ERROR: Too obvious")
}

/* Disabling of Error Propagation from a throwing function using 'try!'*/
let lavarRating = try! checkPW("BigBallerPW")

  • Trailing Closure (syntax)
    • (def)
      • (def) Syntax that allows you to write a closure as a parameter of a function call, following the closing parenthesis of the function's argument.
        • Per Swift Documentation:
          • ”If you need to pass a closure expression to a function as the function’s final argument and the closure expression is long, it can be useful to write it as a trailing closure instead. You write a trailing closure after the function call’s parentheses, even though the trailing closure is still an argument to the function.”
          • *You don’t write the argument label for the closure as part of the function call.
          • “When you use the trailing closure syntax, you don’t write the argument label for the first closure as part of the function call.”
      • *Eliminating of the parentheses when there are no parameters is not specific to the Trailing Closure syntax.
    • Code:
      /* If a closure was the only parameter */
let team = ["LiAngelo", "Laonzo", "Laavar", "LiaMelo"]

let bbb = team.sorted(by: { name1, name2 in
	return name1 > name2 
})

//Trailing Closure syntax:
let bbb = team.sorted() { name1, name2 in
	return name1 > name2 
}

//Parentheses can be eliminated. This is not specific to Trailing Closure syntax.

  • Structs (Structures)
    • (def)
      • Used to store variables of different data types.
      • Used to create custom/complex types with properties and methods.
      • Structs effectively are Tuples without names.
      • A user-defined data type that encapsulates related properties and behaviors into a single value type (Lyu, Lakers, etc.)
      • Swift allows you to define custom types using Structs, Classes, and Enums.
      • Use Structs when you have some fixed data you want to send or receive multiple times. Use tuples when you want to return two or more arbitrary pieces of values from a function.
      • In SwiftUI → Classes: Used extensively for data. Structs: Used extensively for UI design.
      • In UIKit → Structs: Used extensively for data. Classes: Used extensively for UI design.
      • Structs → A Blueprint.
      • Instance → Manifestation of the blueprint instantiated.
        • *Variables are initialized. Objects are instantiated.
    • Syntax (Convention):
      • Structs: Begins capitalized.
      • Things inside struct: camelCase.
    • mutating (keyword)
      • (def) Keyword used to indicate that a method can modify the properties of a value type (struct or enum) from within the method itself.
      • By default, the properties of a value type (in this case a Struct) cannot be modified from within by its instance methods (unless the method follows a mutating keyword).
      • Inside your struct there’s no way of telling whether you’ll be working with a variable struct or a constant struct. So Swift has a simple solution: any time a struct’s method tries to change any properties, you must mark it as mutating.
      • *The mutating method inside the struct:
        • Can only be used on a var property, and not a let property.
        • Cannot be called from an instance that was initialized onto a let constant. It can only be called from an instance that was initialized onto a var variable
    • Properties (get/set)
      • (def) Variable or Constant defined inside a struct/class.
      • 2 kinds of properties:
        • Stored Property
          • (def) Variable or constant that holds a piece of data inside an instance of the struct.
        • Computed Property
          • (def) Property that calculates and returns a value, rather than just store it. Effectively just a function call in a struct. (Can also be computed using conditionals)

            → They can never be declared with a let keyword! Neither for get or set. This is due to computed properties are inherently dynamic in nature. → Provides a getter and an optional setter to indirectly access other properties and values.

            • getter (get): (def) A code block that retrieves and returns the current value of a computed property.
              • read-only computed property: (def) computed property with a getter, but no setter. → the get keyword and its braces can be omitted for simplification.
              • *Declaration of a getter with nothing within code block: var bigBallers: Int { get } (Might use within protocols) → bigBallers in this case can be initialized to be either an Int variable or a computed property.
            • setter (set): (def) A code block that allows the assignment of a new value to the computed property.
              • The value you assign to the computed property (outside the struct), is the value that will be computed in the set block as newValue (the identifier).
              • Without a parameter: Default identifier newValue is used to modify a property.
              • Alternatively, you can use your own identifier (e.g. Pizza) by declaring it as a parameter of the set block.
              • *Declaration of a getter & setter with nothing within code blocks: var bigBallers: Int { get set } (Might use within protocols) → bigBallers in this case can be initialized to be either an Int variable or a computed property.
      • 2 kinds of property observers:
        • (def) Block of code that run when properties change, even if the new value is the same as the property’s current value.
          • It’s for convenience. You could just call a function after updating a property.
        • willSet
          • (def) Called just before the value is changed. It comes with a default parameter that contains the new value, with the default name newValue.
          • Parameter name can be customized. Enter identifier of your choice within parentheses following willSet, and that’s your new identifier.
        • didSet
          • (def) Called immediately after a new value is changed. It comes with a default parameter that contains the old value, with the default name oldValue.
          • Parameter can be customized.
    • Memeberwise Initializer
      • Structs automatically get a default initializer called a memberwise initializer synthesized by the compiler. (Unless you define a custom initializer within the Struct.)
      • memberwise initializer (def): A shorthand way to initialize the member properties of new structure instances. Swift by default generates an initializer using a struct’s property names. Simplifies the process of instantiating.
      • ‘memberwise initializer’ as the compiler will generate it based on the given struct’s members (that is, its stored properties).
      • If all non-private properties within a Struct has values assgined to them all already, it may look like a memberwise initializer is used. But because you gave your properties default values, the initalizer without parameters init () is called instead (since you don't need to assign any properties.).
        • *IAW, if a struct with one property (whether private or non-private) has a value assigned to it already, a memberwise initializer is not used. Instead, it seems to use init(). Just an ‘initializer’. Not ‘memberwise’ or ‘custom’. It requires no parameters, no nothing inside.
        • It’s the reason why this doesn’t throw an error:
          struct BigBaller {
    private var bbbNumber = 3
}
let lavar = BigBaller()

  • Access Control
    • (def) Refers to the Access level modifiers used to set the accessibility of structs/classes/enums/properties/methods/initializers/suscripts.
    • internal: (def) (default) Only your Swift code can read and write the property.
    • open: ?
    • public: (def) Accessible from anywhere.
    • fileprivate: (def) Accessible only within the current Swift file.
    • private: (def) Accessible only within the defined struct/class.

  • static properties/methods
    • (def)
      • Used to define a type-level (stucts/classes) property or method that belongs to the type itself rather than an instance of the type.
        • You can reference static properties inside initializers.
        • Useful for something like an app’s version number.
        • Useful for resolving errors resulting from a property within a struct attempting to read off of another property within a struct/class.
          • (e.g.) If a computed property is assigned to a State variable within the same struct they both are in, it throws an error. To resolve, have the computed property be of a static type. Now, the computed property belongs to the struct itself, rather than the instances of it.
      • So a static property / static method within a struct/class belongs to the struct/class itself, rather than the individual instances of it.
      • *A static method does not require a mutating keyword!
      • On mix-matching static properties/methods with non-static properties/methods:
        • 1) static properties/methods can’t refer to non-static properties/methods.
        • 2) non-static properties/methods can refer to static properties/methods.
          • To refer to a (example) static property from within the struct/class: → Self.maxCharge (Notice capitalization)
          • To refer to a (example) static property from outside the struct/class: → CreditCard.maxCharge (Convention)

    • Classes vs Structs:
      • Similarities:
        1. Requires each properties to have a value, either by way of initializing or assigning default values.
      • Differences:
        1. No memberwise initializer is generated by default.
          • It’s because it’s tricky to implement, given that one class can be based upon another – if a new class (class C) added an extra property it would break all the initializers for C, B, and A.
          • Also: When you decide to change your initializer, you’re doing so yourself, and are therefore fully aware of the implications for any inheriting classes.
          • This requires you to either:

            → Create custom initializer.

            → or Assign default values to all properties.

        1. You can inherit from another classes (make one class build upon functionality in another class), gaining all its properties and methods as a starting point.
        1. Class is a Reference Type as opposed to a Value Type, which is what Structs are. So one instance of a class can be referenced in several places.
          • Any changes you make to one instance will automatically change the other instances.
          • Using a class rather than a struct sends a strong message that you want the data to be shared somehow, rather than having lots of distinct copies.
        1. Classes have deinitializers. Structs don’t.
        1. var and let properties in var and let classes behave differently from var and let structs:
          • Classes:
            • var properties in var classes can be modified.
            • var properties in let classes can be modified. *(1)
            • let properties in var classes cannot be modified.
            • let properties in let classes cannot be modified.
          • Structs:
            • var properties in var structs can be modified (using mutating keyword).
            • var properties in let structs cannot be modified. *(2)
            • let properties in var structs cannot be modified.
            • let properties in let structs cannot be modified.
          • *(1): var properties in let classes can be modified, because when a class is a constant, it means that the class points to a one and only data. Whereas when a struct is a constant, it means that the struct itself is immutable.
          • *(2): var properties in let structs cannot be modified, because when you change a property of a Struct, you are changing the entire struct. (Not ‘just one part’ of a struct.)

    • Deinitializers
      • (def)
        • Special method optionally given to classes that is called immediately before a class’ very last class instance is deallocated.
          • iow, it tells us when a class instance was destroyed.
          • *deallocation: (def) The process of removing an object or instance from memory, making the memory previously occupied by that object available for other uses.
        • With classes, since it’s a reference type, and since there can be multiple class instances that point to the same data, it's difficult to tell when the final class instance pointing to gets deallocated. Which is why deinitalizers are useful.
          • Unlike with structs, where because they’re a data type, once a struct instance is gone, you know it’s really gone. Hence why deinitalizers aren’t necessary for structs.
          • (e.g. If a struct is created inside a method and the methods ends, the struct is destroyed.)
        • Doesn’t take any parameters or return data. Thus, deinit method is without parentheses.
        • They are never called directly, and instead handled by the system.
      • How it works behind the scenes
        • Swift performs something called ARC (automatic reference counting). ARC tracks how many copies of each class instance exists: every time you take a copy of a class instance Swift adds 1 to its reference count, and every time a copy is destroyed Swift subtracts 1 from its reference count. When the count reaches 0 it means no one refers to the class any more, and Swift will call its deinitializer and destroy the object.

  • Protocols
    • (def)
      • What defines a set of methods/properties/other requirements as a requirement, that can then be ‘adopted’ by a class/structure/enumeration to provide an implementation, or that can then be used as a parameter for a function.
        • What conforms to a protocol seems to be referred to by the protocol name. → WindowGroup initializer within a property conforming to Scene is a ‘scene’. → ‘view’ is nothing more than a type conforming to the View protocol.
        • “Like contracts for code”
        • Any type that satisfies the requirements of a protocol ‘conforms’ to that protocol.
        • Anything that adopts a protocol must include all methods and properties inside.
        • Protocols specifies only the ‘minimum’ a conforming type must have, meaning anything adopting a protocol can add additional properties/method as needed.
        • A type can conform to any number of Protocols.
        • If a sublcass adopts a Protocol, the superclass name comes first, followed by protocol names.
      • Inside the protocol itself: properties cannot be assigned default values, and methods cannot have a code block (Only the method name, parameter & return type). → So no braces { } are necessary!
      • A protocol itself can also be a parameter. (Thus any data conforming to the protocol can be used as an argument.)
        • And so the parameter can be anything at all, as long as it contains the required properties/methods specified within the Protocol.
        • Protocol can also be a return value.
    • Protocol Inheritance
      • (def) A protocol inheriting from another protocol.
      • (e.g.) Struct below confroms to protocol Comparable. → Comparable protocol inherits from the Equatable protocol (protocol inheritance!) behind the scenes. → Hence instances of this struct is able to use both the comparison operators & equality operators!

  • Extensions
    • (def)
      • A language construct used to add functionality to any type.
        • * ’functionality’ in this case refers to: Methods, Computed Properties*, Initializers, etc.
        • *Extensions do not let you add Stored Properties.
    • Reasons to use Extensions:
      1. Cleaner code. Extensions are naturally grouped by the data type they are extending, thus making your overall code less messy (as opposed to adding properties/methods in each individual types / having to write lengthier global functions).
      1. Extensions get full access to the type’s (that they are extending) internal data. Meaning the extension has access to properties/methods marked with private acess control.
      1. Enables you to modify values in place. (i.e.) To change a value directly, as opposed to returning a new value.
        • *Naming conventions for methods with different return values:
          • Methods that return a new value → .sort(), .bbb()
          • Methods that returns the value directly changed → .sorted(), .bbbed()
      1. Allows you to use a custom initializer while still retaining Swit’s automatic memberwise initializer.
        • By implementing the custom initializer inside an Extension, Swit does not disable the automatic memberweise initializer.
      1. When you type a dot (.) following a variable you want to use a method created with an Extension on, (quote.) Xcode brings up a list of methods to use, including the ones we added using Extensions. Thus making our extra functionalities easy to find.
    • Conformance grouping & Purpose grouping
      • Conformance grouping
        • (def) Adding a protocol conformance to a type as an extension, adding all the required methods inside that extension.
        • This makes it easier to understand how much code a developer needs to keep in their head while reading an extension.
      • Purpose grouping
        • (def): Creating extensions for specific tasks, making it easier to work with larger types.
        • (e.g.) Using an extention to handle loading/saving of that type.
    • Protocol Extensions
      • (def) Extensions used to add functionality to existing protocols. Any types conforming to the protocols gets them.
      • (e.g.) The commonly used isNotEmpty computed property implemented by way of using a Protocol Extension which extends to the Collection protocol.
        • Example:
        extension Collection {
    var isNotEmpty: Bool {
        isEmpty == false
    }
}
// They're common because it's easier to read rather than saying:
lavarRoom.isEmpty == false
    • *Protocol-Oriented Programming
      • (def) Technique where you list required methods in a protocol, then add default implementations of those inside a protocol extension. → *protocol extension is under notes section ‘Extensions’.
      • Example:
      protocol BigBaller {
		var name: String { get }
}

  • Optionals
    • (def)
      • A type that represents either a value or an absence of a value (nil).
        • Examples:
          • var tweet: String?
          • var tweet: String? = “Neva lost!” (Even with a value assigned, tweet still is an Optional!)
          • var someDictionary = [”Lonzo”: “Pelicans”, “LaMelo”: “Hornets”] var lavarTeam = someDictionary[”Lavar”] (lavarTeam is now an optional string set to ‘nil’.) (It’s the same value as it’d be here: var lavarTeam: String? = nil)
          • var lavar = “" var rings = Int(lavar) // Attempt to convert string to int returns nil. // rings becomes nil, thus rings is now an optional integer. (rings is now the same value as it’d be here: var rings: Int? = nill)
      • Using a value that may be within an optional requires you get it unwrapped.
        • There is a difference between a string optional and a string. They are not the same.
      • To unwrap:
        1. if-let Syntax
        1. Forced Unwrapping
        1. Optional Binding
        1. Optional Chaining
        1. Nil Coalescing
      • Represented by placing ? after a data type (Int?).
        • Any kind of data can be optional. Int/Double/Bool/String, as well as instances of enums/structs/classes.
      • nil: ‘Value’ representing no value.
    • Unwrapping methods
      • Optional binding (if-let)
        • (def) Common unwrapping method which combines creating a condition (if) with creating a temporary constant (let). Unwrapping done to use the value (if there exists one). Code under if block runs if the optional does have a value.
          var number = Int?
number = 9000

if let number = number {
	 
}
          • Called ‘binding’ due to how it ‘binds’ or ‘assigns’ an optional value to a non-optional variable.
          • The value that may be there after unwrapping is only available within the condition’s body. Outside of it, it will still be treated as an unwrapped optional.
          • *The temporary constant is only available within the condition’s body. → This is called ‘shadowing’. Commonly used in optional unwraps.

            → shadowing (def): Declaring of a variable of the same name as an existing variable in a more nested scope (the temporary constant in our context), causing the innermost constant to take precedence over the outer variable within that scope.

      • Optional binding (guard-let-else)
        • (def) Similar to the if-let syntax, except block below guard executes if the optional does not have a value. Unwrapping done to ensure optionals have something inside, and to exit if otherwise.

      • ? (Optional Chaining)
